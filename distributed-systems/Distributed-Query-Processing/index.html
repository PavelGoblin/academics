SECTION 1

What is Distributed Query Processing?

Distributed Query Processing (DQP) is the method of executing a single SQL query across multiple computers (nodes) connected via a network, making distributed data look like one unified database to the user.

Why it exists:

  • Data too big for one machine
  • Parallel execution = faster results
  • Users & data spread globally
  • Fault tolerance (no single point of failure)
  • Organizational autonomy
Analogy: A restaurant manager (coordinator) splits an order among chefs (workers). All chefs cook in parallel. The manager assembles the plate and serves the customer — who never sees the kitchen.

SECTION 2

Architecture Overview

Coordinator (Boss)

  • Parses & optimizes the query
  • Splits work into sub-queries
  • Dispatches to workers
  • Assembles final result

Workers (Team)

  • Store data fragments
  • Execute sub-queries locally
  • Return partial results

All workers run in parallel

SECTION 3

Query Decomposition

Breaking SQL into relational algebra operations — the "planning" phase.

Five basic operations:

  • σ Selection — filter rows
  • π Projection — pick columns
  • ⋈ Join — combine tables
  • ⋃ Union — combine rows
  • − Difference
SELECT e.name, d.dept FROM Emp e, Dept d
WHERE e.did = d.did AND e.sal > 50000
→ σsal>50000(Emp)   πdept,did(Dept)   then join

SECTION 3 (cont.)

Four Detailed Steps of Decomposition

1. Normalization → 2. Analysis → 3. Redundancy Elimination → 4. Rewriting

Normalization: WHERE clause → standard form

  • CNF: AND of ORs — (A∨B) ∧ (C∨D)
  • DNF: OR of ANDs — each term = one sub-query + UNION

Analysis: Reject incorrect queries
Type errors & semantic errors caught early

Redundancy Elimination: Simplify predicates

WHERE (NOT(pos='striker') AND (pos='striker' OR pos='keeper') AND NOT(pos='keeper')) OR ...
↓ simplifies to ↓
WHERE pid = p.pid AND name = 'Muller'

Rewriting: Build algebraic tree, then push σ & π down, reorder joins

SECTION 5

Global Query Optimization

Find the cheapest plan — minimize data sent over the network.

Cost Model: Total Cost = CPU + I/O + Network
Network > I/O > CPU. Moving 1MB over network costs more than processing 10MB locally.

Join Ordering: different orders = drastically different costs

Emp 10k rows (1MB) @ Site 1 • Dept 100 rows (3.5KB) @ Site 2
Cheapest: Ship Dept → Site 1, join, ship result → Site 3 = 403,500 bytes
vs: Ship both to Site 3 = 1,003,500 bytes

Key decisions:

  • Which table to ship where?
  • Use semi-joins or ship-whole?
  • Use Bloom filters?
  • Order of joins (dynamic programming)
Goal: Minimize data movement. Network is the bottleneck.

SECTION 5 (cont.)

Semi-Join Reduction Strategy

Instead of shipping an entire table, send only join column values:

  1. πjoin_attr(R) → tiny R'
  2. Ship R' to S's site
  3. S' = S ⬆ R' (keep matching rows)
  4. Ship reduced S' back
  5. Complete join R ⋈ S'
Analogy: Bring only the VIP list to the party venue, not the entire guest book.
join(R) = R' tiny R' S' = S ⬆ R' reduced S' R ⋈ S' at Coordinator Method Data Transferred Winner? Ship-Whole |R| = large ✗ Semi-Join |π(R)| + |S'| = small ✓ Bloom Filter |BF| + |S'| = smallest ✓✓

SECTION 5 (cont.)

Local Query Optimization

After the global plan is chosen, each site still optimizes its own part:

  • Choose index scan vs full table scan
  • Reorder local joins
  • Push σ & π down the tree
  • Use local statistics (index cardinality)
Why it matters: A 10× speedup at each worker → 10× faster overall query.
The 4 Phases of DQP Optimization:
1. Query Decomposition → logical algebra
2. Data Localization → fragment assignment
3. Global Optimization → minimize network cost
4. Local Optimization → maximize per-node speed

SECTION 7

Data Fragmentation Types

Horizontal (by rows)

Each fragment = all columns, subset of rows

Emp_N = σregion='N'(Emp)
Emp_S = σregion='S'(Emp)

Reconstruct: Emp_N ∪ Emp_S

Vertical (by columns)

Each fragment = all rows, subset of columns. Key column replicated.

Emp_a = πeid,name(Emp)
Emp_b = πeid,salary(Emp)

Reconstruct: Emp_a ⋈eid Emp_b

Mixed (Hybrid)

Combine both: e.g., vertical split first, then horizontal on each piece. Most flexible, most complex.

SECTION 8

Result Assembly

Coordinator collects partial results and assembles the final answer.

Assembly by query type:

  • SELECT: UNION all partial results
  • JOIN: Coordinator joins intermediate results
  • AGGREGATE: Re-aggregate (sum the sums, etc.)
  • ORDER BY: Merge sorted runs (N-way merge sort)
Counting total employees:
Worker N: 200 • Worker S: 350 • Worker E: 180 • Worker W: 270
Coordinator: 200+350+180+270 = 1000

SECTION 9

Failure Handling

In distributed systems, failures are inevitable — the system must detect and recover.

FailureImpactSolution
Node crashPartial results lostRe-route to replica; or retry later
Network partitionCoordinator can't reach workersTimeout; work with reachable nodes only
Coordinator failureEntire query lostStandby coordinator + checkpointing
Slow node (straggler)Query delayedSpeculative execution (hedged requests)
Data corruptionWrong result silentlyChecksums; redundant execution

WRAP-UP

Key Takeaways

  • DQP makes distributed data look like a single database
  • Coordinator plans & orchestrates; workers execute
  • Network cost dominates — minimize data movement
  • Fragmentation: horizontal (rows), vertical (columns), mixed
  • Semi-joins & Bloom filters reduce network traffic
  • 4 optimization phases: decompose → localize → global → local
  • Failure handling via heartbeat, retry, replication, speculation
  • Result assembly depends on query type (UNION / re-aggregate / merge sort)
Distributed Query Processing = making the distributed world feel like a single database.
