What is Distributed Query Processing?

Distributed Query Processing is the process of executing database queries in a distributed database system where data is stored across multiple network-connected sites.

💡 Simple Explanation:

Employee data in Dhaka + Department data in Chittagong = System accesses both → processes → combines results

What is a Distributed Database?

A collection of logically related databases distributed over different physical locations and connected through a communication network. To the user, it appears as a single database.

🗄️
Data Distribution
Data is stored in different physical sites across the network
🌐
Network Communication
Sites communicate through a reliable communication network
🏢
Local Autonomy
Each site can manage its own data independently
✨
Transparency
Users don't need to know where data is physically stored

Why DQP is Necessary

❌ Without DQP

  • ⚠️ High network traffic
  • 🐌 Slow query execution
  • 📉 Decreased system performance
  • 💸 Unnecessary data transfer
  • 🔄 Poor resource utilization

✅ With DQP

  • ✅ Reduces unnecessary data transfer

  • 🎯 Optimizes resource usage

  • 📈 Improves scalability

Main SQL Query

SELECT E.Name
FROM Employee E, Department D
WHERE E.Dept_ID = D.Dept_ID
  AND D.Location = 'Dhaka';
🎯
Objective:

Retrieve names of employees working in departments located in Dhaka

Processing Step 1: Site B Query

📍 Query at Site B

SELECT Dept_ID
FROM Department
WHERE Location = 'Dhaka';

✅ Output

D1
💡 Key Benefit: Only D1 is sent to Site A instead of the whole Department table → Reduces communication cost

Semi-Join Technique

Optimization method that reduces data transfer between sites

❌ Without Semi-Join

  • 📦 Full Department table sent to Site A
  • 🐌 Heavy network traffic (20 rows)
  • 💸 High communication cost

✅ With Semi-Join

  • 📤 Only matching Dept_IDs sent (2 values)
  • ⚡ Lightweight transfer (2 rows)
  • 💰 90% reduction in network load
✅ Lower cost | ⚡ Faster execution | 📉 Reduced traffic

Query Shipping vs Data Shipping

📤 Query Shipping

Query is sent to the site where data exists

  • ✅ Less data transfer
  • ✅ Processing done at data site
  • ✅ Small result returns
📤 → 🗄️ → 📤

🗄️ Data Shipping

Data is transferred to query site for processing

  • 🔄 Data moves to user site
  • 📊 Good for complex local ops
  • 💡 Centralized processing
🗄️ → 📤 → 💻

Disadvantages of DQP

🌐
High Network Dependency

🔄
Synchronization Problems
Maintaining data consistency across sites is challenging
🔒
Security Issues
Data transfer over network creates security vulnerabilities

Major Challenges in DQP

📡
Communication Overhead

💥
Site Failure
Failure of one server may interrupt entire query execution
🧩
Data Fragmentation
Managing fragmented data across sites adds complexity
🔁
Distributed Deadlock
Transactions across multiple sites may cause deadlocks
