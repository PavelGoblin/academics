Lab 01
Drawing a Straight Line using C Program
1.1   Objective

To write a C program that draws a straight line on the screen using the basic parametric line equation, utilizing the putpixel() function available in BGI graphics (Turbo C / DevC++ with WinBGIm) or equivalent OpenGL primitive.

1.2   Theory

A line segment between two points P₁(x₁, y₁) and P₂(x₂, y₂) can be described using the parametric form:

x(t) = x₁ + t · (x₂ − x₁)
y(t) = y₁ + t · (y₂ − y₁)
where t ∈ [0, 1], sampled at intervals of 1 / max(|Δx|, |Δy|)

The number of steps is set to max(|Δx|, |Δy|) to ensure no pixel gaps along either axis. At each step, the floating-point coordinates are rounded to the nearest integer before calling putpixel().

1.3   Algorithm Steps
  1. Input two endpoints: (x1, y1) and (x2, y2)
  2. Compute dx = x2 − x1 and dy = y2 − y1
  3. Compute steps = max(|dx|, |dy|)
  4. Compute xInc = dx / steps and yInc = dy / steps
  5. Initialize x = x1, y = y1
  6. Loop i from 0 to steps: plot pixel at (round(x), round(y)), then x += xInc, y += yInc
1.4   Source Code
#include <stdio.h>
#include <graphics.h>
#include <math.h>

void drawLine(int x1, int y1, int x2, int y2) {
    int dx = x2 - x1;
    int dy = y2 - y1;
    int steps = (abs(dx) > abs(dy)) ? abs(dx) : abs(dy);
    float xInc = (float)dx / steps;
    float yInc = (float)dy / steps;
    float x = x1, y = y1;
    int i;
    for (i = 0; i <= steps; i++) {
        putpixel((int)round(x), (int)round(y), WHITE);
        x += xInc;
        y += yInc;
    }
}

int main() {
    int gd = DETECT, gm;
    initgraph(&gd, &gm, "");
    drawLine(50, 50, 350, 200);
    getch();
    closegraph();
    return 0;
}

Lab 02
Line Drawing using DDA Algorithm
2.1   Objective

To implement the Digital Differential Analyzer (DDA) algorithm in C for rasterizing a line segment between two given endpoints on a pixel grid.

2.2   Theory

The DDA algorithm is based on the differential equation of a line. Given slope m = dy/dx, the algorithm samples at unit intervals along the dominant axis and computes the other coordinate by accumulation:

Case |dx| ≥ |dy| (shallow): x increments by 1, y += m (slope)
Case |dy| > |dx| (steep): y increments by 1, x += 1/m

Generalized: steps = max(|dx|,|dy|), xInc = dx/steps, yInc = dy/steps

DDA improves on the brute parametric method by always incrementing the dominant axis by exactly 1, reducing the number of float operations. However, cumulative float rounding errors can accumulate over long lines.

2.3   Algorithm Steps
  1. Read endpoints (x1, y1) and (x2, y2)
  2. Calculate dx = x2 − x1, dy = y2 − y1
  3. Set steps = max(|dx|, |dy|)
  4. xInc = dx / steps  ·  yInc = dy / steps
  5. Initialize x = x1, y = y1
  6. For i = 0 to steps: putpixel(round(x), round(y)); x += xInc; y += yInc
2.4   Source Code
#include <stdio.h>
#include <graphics.h>
#include <math.h>

void DDA(int x1, int y1, int x2, int y2) {
    int dx = x2 - x1;
    int dy = y2 - y1;
    int steps = (abs(dx) > abs(dy)) ? abs(dx) : abs(dy);
    float xInc = (float)dx / steps;
    float yInc = (float)dy / steps;
    float x = x1, y = y1;
    int i;
    for (i = 0; i <= steps; i++) {
        putpixel((int)round(x), (int)round(y), CYAN);
        x += xInc;
        y += yInc;
    }
}

int main() {
    int gd = DETECT, gm;
    initgraph(&gd, &gm, "");
    DDA(40, 200, 360, 60);
    getch();
    closegraph();
    return 0;
}

Lab 03
Line Drawing using Bresenham's Algorithm
3.1   Objective

To implement Bresenham's Line Drawing Algorithm in C, which uses only integer arithmetic (no floating-point operations) to determine pixel positions along a rasterized line.

3.2   Theory

Bresenham's algorithm (1965) uses a decision parameter p derived from the line equation to decide, at each step, whether the next pixel should stay at the same y level or increment y by 1. The key insight is that all quantities can be maintained as integers:

Initial decision parameter: p₀ = 2·dy − dx

If p < 0: plot (x+1, y), update p = p + 2·dy
If p ≥ 0: plot (x+1, y+1), update p = p + 2·dy − 2·dx

For general octants: use sign(dx) and sign(dy) to handle all directions.

This algorithm runs faster than DDA because it avoids all floating-point operations and uses only addition and comparison — operations that are very fast on hardware.

3.3   Algorithm Steps (General — All Octants)
  1. Compute dx = |x2−x1|, dy = |y2−y1|
  2. Set sx = sign(x2−x1), sy = sign(y2−y1)
  3. Initialize err = dx − dy
  4. Loop: plot (x, y); if reached endpoint, break
  5. e2 = 2·err; if e2 > −dy: err −= dy; x += sx
  6. if e2 < dx: err += dx; y += sy
3.4   Source Code
#include <stdio.h>
#include <graphics.h>
#include <stdlib.h>

void Bresenham(int x1, int y1, int x2, int y2) {
    int dx = abs(x2 - x1);
    int dy = abs(y2 - y1);
    int sx = (x1 < x2) ? 1 : -1;
    int sy = (y1 < y2) ? 1 : -1;
    int err = dx - dy;
    int e2;

    while (1) {
        putpixel(x1, y1, YELLOW);
        if (x1 == x2 && y1 == y2) break;
        e2 = 2 * err;
        if (e2 > -dy) { err -= dy; x1 += sx; }
        if (e2 <  dx) { err += dx; y1 += sy; }
    }
}

int main() {
    int gd = DETECT, gm;
    initgraph(&gd, &gm, "");
    Bresenham(30, 220, 380, 50);
    getch();
    closegraph();
    return 0;
}

3.7   Result & Discussion

Bresenham's algorithm produces a visually identical line to DDA, but uses only integer arithmetic throughout. The decision parameter err is updated by simple addition and subtraction. The algorithm handles all eight octants through the sign variables sx and sy. Performance advantage is significant in hardware implementations such as GPU rasterizers.

3.8   Conclusion

Bresenham's algorithm is the gold standard for line rasterization in computer graphics. Its exclusive use of integer arithmetic makes it both fast and accurate. Modern GPU rasterizers are fundamentally based on variations of this approach.

Lab 04
Geometric Translation (2D Transformation)
4.1   Objective

To implement geometric translation — a fundamental 2D affine transformation — in C, which repositions every vertex of a shape by a fixed offset vector (Tx, Ty).

4.2   Theory

Translation moves a geometric object from one position to another without rotation or scaling. For each point P(x, y) in the object, the translated point P'(x', y') is given by:

x' = x + Tx
y' = y + Ty

Homogeneous matrix form (3×3):
[ x' ] [ 1 0 Tx ] [ x ]
[ y' ] = [ 0 1 Ty ] [ y ]
[ 1 ] [ 0 0 1 ] [ 1 ]

The homogeneous coordinate representation is particularly important because it unifies all affine transformations (translation, rotation, scaling, shearing) into single matrix multiplications, enabling efficient composition of transforms.

4.3   Algorithm Steps
  1. Define the original polygon vertices as arrays x[] and y[]
  2. Input translation values Tx and Ty
  3. Draw the original shape using line drawing functions
  4. For each vertex i: x[i] = x[i] + Tx; y[i] = y[i] + Ty
  5. Draw the translated shape using updated coordinates
  6. Optionally draw arrows from original to translated vertices
4.4   Source Code
#include <stdio.h>
#include <graphics.h>

void drawPolygon(int x[], int y[], int n, int color) {
    int i;
    setcolor(color);
    for (i = 0; i < n - 1; i++)
        line(x[i], y[i], x[i+1], y[i+1]);
    line(x[n-1], y[n-1], x[0], y[0]);
}

void translate(int x[], int y[], int n, int tx, int ty) {
    int i;
    for (i = 0; i < n; i++) {
        x[i] += tx;
        y[i] += ty;
    }
}

int main() {
    int gd = DETECT, gm;
    initgraph(&gd, &gm, "");

    int x[] = {100, 200, 200, 100};
    int y[] = {100, 100, 200, 200};
    int n = 4;
    int tx = 150, ty = 80;

    drawPolygon(x, y, n, WHITE);   // draw original
    translate(x, y, n, tx, ty);    // apply translation
    drawPolygon(x, y, n, CYAN);    // draw translated

    outtextxy(110, 210, "Original");
    outtextxy(260, 290, "Translated");

    getch();
    closegraph();
    return 0;
}

4.7   Result & Discussion

The translation program successfully repositions all vertices of the polygon by the specified (Tx, Ty) offset. The shape's geometry — angles, side lengths, and orientation — remains completely unchanged. Using homogeneous coordinates enables translation to be expressed as a matrix multiplication, which is essential for composing multiple transformations efficiently in graphics pipelines (e.g., model-view-projection matrices in OpenGL).

4.8   Conclusion

Translation is the simplest of the geometric transformations, yet it is foundational to all 2D and 3D graphics. Understanding translation in matrix form prepares the ground for rotation, scaling, and composite transformations studied in advanced computer graphics. The implementation is straightforward: a simple loop adding Tx and Ty to each coordinate pair.

Analysis
Comparative Analysis of Line Drawing Algorithms
5.1   Time & Space Complexity
Algorithm Time Complexity Space Complexity Arithmetic
ParametricO(n)O(1)Float (+, ×, ÷)
DDAO(n)O(1)Float (+, ÷)
BresenhamO(n)O(1)Integer (+, −, ≪)

where n = max(|dx|, |dy|) is the number of pixel steps. All three algorithms are linear in the dominant axis length. Bresenham's constant-factor advantage comes from replacing float division/multiplication with integer addition and bit-shift operations, making it ~2–3× faster in practice on CPU and the preferred choice in hardware rasterizers.

5.3   Cumulative Error Analysis

Both Parametric and DDA accumulate floating-point rounding errors as ε · n where ε ≈ 1×10⁻⁷ (single precision). After 10⁶ steps, worst-case drift reaches ~0.1 pixels — negligible for display purposes but problematic for precise CAD/CAM rasterization. Bresenham's integer-only accumulator err is exact: the chosen pixels always minimize the true Euclidean distance to the ideal line, provably optimal.

5.4   Decision Parameter Derivation (Bresenham)

The decision parameter pk at step k is derived from the difference between the two candidate pixel distances from the true line:

pk = 2·dy · xk − 2·dx · yk + 2·dy + dx · (2·y₁ − 1) − 2·dy · x₁

Simplified recurrence (for slope 0 < m < 1):
p₀ = 2·dy − dx
pk+1 = pk + 2·dy  — if pk < 0
pk+1 = pk + 2·dy − 2·dx  — if pk ≥ 0

This recurrence eliminates multiplication entirely, requiring only addition and comparison at each step — the hallmark of Bresenham's efficiency.

5.5   Extension: Anti-Aliased Lines (Wu's Algorithm)

Bresenham's algorithm plots exactly one pixel per column (or row). For smoother visual appearance, Xiaolin Wu's algorithm (1991) draws two pixels per step with intensity proportional to coverage area — known as anti-aliasing. The brightness of each pixel is computed as:

End brightness = fractional part of the computed y-coordinate (or x)
Adjacent pixel brightness = 1 − end brightness

Wu's algorithm trades ≈2× more pixel writes for significantly reduced staircasing (jaggies). It is commonly used in modern 2D rendering libraries (e.g., Cairo, Skia) where smooth lines are required.

Transform
2D Affine Transformations: A Unified View
6.1   Transformation Matrices

All standard 2D affine transformations can be expressed as 3×3 homogeneous matrices. Composition is achieved by simple matrix multiplication:

TransformationMatrix (3×3)Inverse
Translation[1 0 Tx; 0 1 Ty; 0 0 1]Translate by (−Tx, −Ty)
Rotation (θ)[cosθ −sinθ 0; sinθ cosθ 0; 0 0 1]Rotate by −θ
Scale (Sx, Sy)[Sx 0 0; 0 Sy 0; 0 0 1]Scale by (1/Sx, 1/Sy)
Shear (X, Y)[1 Shx 0; Shy 1 0; 0 0 1]Shear by (−Shx, −Shy)
Reflection (X-axis)[1 0 0; 0 −1 0; 0 0 1]Same (involution)

The homogeneous form allows any sequence — e.g., Translate · Rotate · Scale — to be pre-multiplied into a single 3×3 matrix, eliminating per-vertex overhead. This is the mathematical foundation of the model-view-projection pipeline in OpenGL and DirectX.

6.2   Composition Order Matters
Rotate then Translate vs Translate then Rotate produce different results.

Example: Rotate a square 45°, then translate — the square moves along the rotated axes.
Translate first, then rotate — the square orbits around the origin.

In computer graphics, the standard convention is Scale → Rotate → Translate (SRT) when building a model matrix, applied right-to-left: P' = T · R · S · P.

Use
Practical Applications in Computer Graphics
7.1   Line Drawing
  • Vector graphics rendering — SVG, PDF, PostScript rasterizers use Bresenham-based algorithms to convert vector paths to pixel displays
  • CAD software — AutoCAD, SolidWorks, and KiCad rely on integer line drawing for wireframe previews at interactive frame rates
  • Game engines — Early 2D engines (e.g., Doom, Wolfenstein 3D) used Bresenham for raycasting and wall rendering
  • Printer drivers — Laser and inkjet printer firmware implements Bresenham for nozzle/pin control
  • LCD/OLED controllers — Embedded display drivers on microcontrollers often have Bresenham in silicon
7.2   Geometric Transformations
  • Computer animation — Translation is the fundamental operation for keyframe interpolation; position curves (splines) generate intermediate Tx, Ty values per frame
  • GUI systems — Window managers apply translation matrices for window dragging, scrolling, and viewport panning
  • Robotics & vision — Image registration and stitching use 2D affine transforms (including translation) to align overlapping photos
  • Geographic Information Systems (GIS) — Map panning and coordinate system transformations (e.g., from WGS84 to screen coordinates) rely on translation and scaling
