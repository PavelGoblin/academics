🎁

Cookie

Data stored on the client. Browser sends cookie with every request to the domain.

// Client-side storage
Set-Cookie: session_id=abc123; Secure; HttpOnly
🔒

Session

Data stored on the server. Only a Session ID travels to the client via cookie.

// Server-side storage
redis.set("session:abc123", user_data)
📚

Library Card

The librarian issues a membership card. You write your name on it and present it each time you borrow a book. The card stays with you (the client). If lost, the librarian no longer recognizes you — the data goes with it.

🏦

Bank Locker

The bank gives you a locker key (Session ID). Your valuables stay in the bank's vault (server). You carry the key, but the contents remain secure. Even if the key is lost, the vault stays locked.

💡 Key Insight: LOCATION is the fundamental difference — client-side (cookie) vs server-side (session). This single architectural choice determines everything else: security, size limit, lifetime control, and blocking behavior.

🎁

Cookie Characteristics

Client-side, ~4KB limit, visible, less secure, configurable lifetime, can be blocked, no server cost. Best for UI preferences, tracking, and non-sensitive data.

🔒

Session Characteristics

Server-side, unlimited, hidden, more secure, server-controlled lifetime, cannot be blocked, uses server resources. Best for auth tokens, payments, personal data.

⚠ Rule of Thumb: If the data fits on a cookie AND is not sensitive — use a cookie. If the data is sensitive OR exceeds 4KB — use a session.
Question 2

Facebook & Google

How two tech giants manage billions of sessions and cookies — with fundamentally different consistency strategies at planetary scale.

🎁

Cookies

c_user Encrypted User ID
xs Session + CSRF token
datr Browser fingerprint
presence Messenger online status

All auth cookies: Secure HttpOnly SameSite=Lax

⚙

Infrastructure

TAO: Custom distributed data store — eventual consistency
Memcache: Ultra-fast session caching layer
Global replication: US, Europe, Asia data centers
30-day session timeout • Login alerts • 2FA • Device tracking

Facebook serves 3+ billion users using eventual consistency (TAO) — data is written fast and propagated asynchronously. Your feed may not be identical across data centers, but it's available everywhere instantly.
🎁

Cookies

SID / HSID Cross-service auth for *.google.com
One login powers: Gmail, YouTube, Drive, Maps, Calendar, Photos, Android
Access Token Short-lived (~1 hour)
Refresh Token Long-lived, revocable
Passkeys (FIDO2): Passwordless biometric login

⚙

Infrastructure

Spanner: Globally distributed SQL database — strong consistency
ACID transactions across all regions
99.999% availability SLA
OAuth 2.0 + OpenID Connect for delegated auth
Advanced Protection Program for high-risk users

Google serves 1+ billion users using strong consistency (Spanner) — ACID transactions across all data centers. Your Gmail inbox looks the same whether you're in New York or Tokyo.

⚡ Facebook — TAO

  • Consistency: Eventual
  • Speed: Very fast writes
  • Data: May lag across regions (~seconds)
  • Scale: 3B+ users
  • Architecture: Social graph + cache
  • Best for: Feed, messaging, likes

🔍 Google — Spanner

  • Consistency: Strong (ACID)
  • Speed: Slightly slower writes
  • Data: Always consistent globally
  • Scale: 1B+ users
  • Architecture: SQL + TrueTime API
  • Best for: Email, cloud storage, payments
⚠ Fundamental Trade-off: Facebook chose eventual consistency (TAO) for maximum write speed and global scale. Google chose strong consistency (Spanner) for data accuracy across all services. Neither is wrong — they optimized for different products. Facebook needs you to post a status instantly; Google needs your bank transaction to be correct everywhere.
Question 3

"Remember Me" Implementation

Step-by-step architectural deep-dive: how persistent authentication works — from login credentials to cryptographic token rotation.

1
User submits email + password with "Remember Me" checkbox checked
2
Server validates credentials against hashed password in database
3
Server generates a cryptographically random 256-bit token
4
Token is SHA-256 hashed and stored in database — NEVER store the raw token!
5
Original (unhashed) token is sent as a persistent cookie with Max-Age=30 days
Set-Cookie: remember_token=abc123...; Max-Age=2592000; Secure; HttpOnly; SameSite=Lax; Path=/
6
User closes browser — cookie persists because Max-Age is set (unlike session cookies that die on close)
7
Next visit: browser automatically sends the remember_token cookie with the request
8
Server hashes the received token and looks up the hash in the database
9
If valid and not expired — auto-login the user by creating a new session
10
TOKEN ROTATION: Delete old token from DB, generate a new 256-bit token, hash it, store the hash, set a new cookie. This limits damage if a token is ever stolen — it works only until the real user visits the site and triggers rotation.
🛡

Security Best Practices

✓ SHA-256 hash tokens before storage
✓ Use Secure HttpOnly SameSite=Lax
✓ Token rotation on every use
✓ Short expiry (30 days)
✓ Rate-limit token validation

⚠

Common Mistakes

✗ Storing raw token in DB
✗ No token rotation
✗ Very long expiry (1 year+)
✗ Missing Secure/HttpOnly flags
✗ Predictable token generation

⚠ CRITICAL: Token rotation (step 10) is the single most important security practice in "Remember Me" implementations. It ensures that even if a persistent token is stolen via XSS or leaked traffic, it becomes invalid the next time the legitimate user visits the site. Always rotate on every use.
Question 4

Logout from Specific Device

How selective session termination works — terminate only the compromised device without affecting any other active sessions.

💡

Core Principle

Each device is issued a UNIQUE Session ID (SID). The server maintains a session store mapping SID to user data. Since each device's SID is independent, the server can selectively invalidate individual sessions.

Scenario: Your phone is lost/stolen. You log into a desktop browser, go to Security Settings, and terminate only the phone session. Laptop and desktop sessions continue unaffected.

🛠

Implementation Options

✓ A. DB Flag — Set active=FALSE (Recommended)
✓ B. Delete Record — DELETE FROM sessions WHERE id='...'
⚠ C. Blacklist — Add SID to revoked list (adds latency)
✗ D. Client-side only — Deleting cookie on lost device is NOT sufficient

🔧 Workflow

  • 1. User navigates to Security → Active Sessions
  • 2. Dashboard displays all devices with metadata:
    "iPhone 15 — New York — Active 2 min ago"
    "Chrome 125/Win — San Francisco — 1 hr ago"
    "Safari/macOS — Tokyo — 2 days ago"
  • 3. User clicks "Log Out" next to the iPhone entry
  • 4. Server executes: DELETE FROM sessions WHERE user_id=42 AND device_id='iphone_15'
  • 5. iPhone's next request gets 401 Unauthorized
  • 6. Laptop and desktop sessions remain fully active

⚠ Why Client-Only Fails

  • Clearing the cookie on the lost device only works if that device makes a request
  • If the device is offline (stolen, turned off), the cookie is never cleared
  • An attacker with physical access can still use the stored session
  • The session remains valid on the server indefinitely
  • Server-side invalidation is the only reliable approach:
    ✓ Immediate effect worldwide
    ✓ Works even if device is offline
    ✓ Independent per session
💡 Key Takeaway: Each device = unique SID. Server-side deletion = immediate, global, irreversible. Client-side cookie clearing on a lost/offline device is useless. Always implement server-side session management with a visible "Active Sessions" dashboard.
Question 5

Theme: Cookie or Session?

Should a user's dark/light mode preference be stored in a cookie or a server-side session? Click the toggle below to try a live implementation.

1

Non-Sensitive Data

Theme preference has zero security risk. It's a "T-shirt color, not an ATM PIN." No passwords, no personal information — just a CSS display preference.

2

Needs Persistence

Session cookies delete on browser close. A user would lose their dark mode preference every single time they restart the browser. Persistent cookie (Max-Age=1yr) survives restarts and reboots.

3

Client-Side = Faster

Cookie: JS reads → applies CSS class → <1ms
Session: HTTP request → server reads DB → renders HTML → 50-500ms + flash of unstyled light mode on every page load

Live Interactive Demo

This toggle uses a persistent cookie to remember your theme preference across browser sessions. Close and reopen the page — the preference persists.

document.cookie = "theme=light; max-age=31536000; path=/"

💡 Golden Rule: Use cookies or localStorage for UI preferences (theme, language, layout, font size). Use server sessions for sensitive data (auth tokens, payment info, personal data). When in doubt, choose the more secure option — sessions.

Summary

All 5 Questions — Answered

Q1: Cookie vs Session

Cookie = client storage (browser, ~4KB, visible) | Session = server storage (Redis/DB, unlimited, hidden). Location determines everything: security, size, lifetime, blocking.

Q2: Facebook & Google

Facebook uses TAO + Memcache (eventual consistency for 3B+ users). Google uses Spanner + OAuth 2.0 (strong ACID consistency for 1B+ users). Different trade-offs for different products.

Q3: Remember Me

Hash token (SHA-256) → store hash in DB → persistent cookie (30-day Max-Age) → TOKEN ROTATION on every use — the most critical security practice.

Q4: Device Logout

Each device = unique Session ID. Server deletes/invalidates specific SID. Client-only clearing on a lost/offline device is insufficient — always invalidate server-side.

Q5: Theme Storage

Use COOKIE or localStorage. NOT server session. Non-sensitive, needs persistence across restarts, client-side is instant, no server load, SPA-friendly.

💡 Final Best Practices: Always use Secure HttpOnly SameSite=Lax on authentication cookies. Hash tokens before DB storage. Rotate tokens on every use. Give users a visible session management dashboard. When in doubt about where to store data, choose the more secure option — sessions.
